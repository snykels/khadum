📌 تعليقات على كل نقطة
1. ✅ أعمدة text بدل jsonb — قرار صحيح
اتركها text. لا تحاول تحويلها الآن. التحويل يحتاج:

ALTER TABLE admin_roles 
  ALTER COLUMN permissions TYPE jsonb 
  USING permissions::jsonb;


وهذا يفشل لو أي صف فيه JSON غير صالح.
خطة لاحقة (migration منفصلة، ليس الآن):
	•	التحقق أن كل القيم JSON valid أولاً
	•	migration في وقت هادئ
	•	backup قبل وبعد
في lib/schema.ts الآن:

permissions: text('permissions').notNull().default('[]'),


وفي طبقة التطبيق اعمل type wrapper:

// lib/types/admin.ts
export type AdminRole = Omit<InferSelectModel<typeof adminRoles>, 'permissions'> & {
  permissions: string[]; // parsed
};

export function parseAdminRole(row: InferSelectModel<typeof adminRoles>): AdminRole {
  return { ...row, permissions: JSON.parse(row.permissions || '[]') };
}


2. ✅ settings composite PK — استخدم بالضبط ما اقترحت

(t) => ({
  pk: primaryKey({ columns: [t.ns, t.key] }),
})


3. ⚠️ FKs — ملاحظة مهمة على CASCADE
email_verifications.application_id → freelancer_applications.id (CASCADE) — هذا سليم منطقياً. حذف التقديم يحذف رموز التحقق. لكن وثّقه بتعليق:

applicationId: integer('application_id')
  .references(() => freelancerApplications.id, { onDelete: 'cascade' }),
// ملاحظة: CASCADE مقصود — حذف التقديم يحذف OTPs المرتبطة


الأربعة الأخرى NO ACTION — اترك NO ACTION صراحةً (لا تغيّرها لـ SET NULL أو CASCADE).
4. ⚠️ Index مكرر في email_verifications
اتركه موجوداً الآن. لا تحذفه في هذه المرحلة.
السبب: إذا حذفته من الـ schema، db:push سيسقطه من DB، وهذا destructive change — يخالف قواعدنا لهذه المرحلة.
نظّفه في migration مستقبلية بعنوان:

chore(db): remove duplicate index on email_verifications.token


في lib/schema.ts عرّف الاثنين:

// UNIQUE constraint
token: varchar('token', { length: 255 }).notNull().unique(),

// Index منفصل (مكرر لكن موجود في DB حالياً)
(t) => ({
  tokenIdx: index('email_verifications_token_idx').on(t.token),
})


5. ✅ UNIQUE constraints — عرّفها صراحةً
استخدم .unique() inline أو uniqueIndex في الـ callback. المهم ما تنسى واحد.
6. ✅ email_templates نظيف — ممتاز
7. ✅ serial integer — التزم بالنمط
8. ✅ لا CHECK constraints — سليم

⚠️ ملاحظة حرجة: strict: true غير مدعوم
كلامك صحيح — في drizzle-kit الأحدث الخيار اسمه مختلف.
الحل البديل:

// drizzle.config.ts
import type { Config } from 'drizzle-kit';

export default {
  schema: './lib/schema.ts',
  out: './drizzle',
  dialect: 'postgresql',
  dbCredentials: {
    url: process.env.DATABASE_URL!,
  },
  verbose: true,
  // strict مو مدعوم — نستخدم manual safety instead
} satisfies Config;


التعويض عن غياب strict:

# دائماً استخدم هذه السلسلة بدل db:push مباشر:

# 1. Backup
pg_dump $DATABASE_URL > .local/backups/before_push_$(date +%s).sql

# 2. Generate diff (يعرض ماذا سيحدث)
npx drizzle-kit generate

# 3. راجع الملف SQL المُنشأ يدوياً
cat drizzle/[latest].sql

# 4. لو آمن → push
npx drizzle-kit push


أضف script في package.json:

{
  "scripts": {
    "db:backup": "mkdir -p .local/backups && pg_dump $DATABASE_URL > .local/backups/backup_$(date +%Y%m%d_%H%M%S).sql",
    "db:diff": "npx drizzle-kit generate",
    "db:push:safe": "npm run db:backup && npx drizzle-kit push",
    "db:studio": "npx drizzle-kit studio"
  }
}


من الآن: استخدم db:push:safe فقط. ممنوع db:push مباشر.
